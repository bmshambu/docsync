# Smart RAG(DocSync Agent) — GraphRAG Knowledge Agent for RFP Documents

A local, privacy-first knowledge agent that turns a folder of RFP documents (PDF, DOCX, PPTX) into a queryable knowledge graph — powered entirely by **Claude Cowork** with no external API calls.

Built on the GraphRAG architecture (inspired by Microsoft's GraphRAG paper): entities and relationships are extracted from documents, organised into a NetworkX knowledge graph, and clustered into communities using the Louvain algorithm. Claude acts as the intelligence layer for extraction and querying; Python scripts handle only deterministic file operations.

---

## Architecture overview

```
rfp_data/              ← Drop your RFP documents here
    ↓  [rfp-data-prep skill]
extracted_text/        ← Plain text per document
chunks/                ← 400-word chunks with page + section metadata
    ↓  [Claude extracts]
graph/
  entities.json        ← 14 entity types (client, standard, regulator, …)
  relationships.json   ← Typed edges between entities (requires, governed_by, …)
    ↓  [build_graph.py]
  community_map.json   ← Louvain community clusters
    ↓  [rfp-community-summarizer skill]
  communities/
    community_00.md    ← Plain-English summary per cluster
    community_01.md
    …
    ↓  [rfp-query-agent skill]
User question → query_graph.py → Claude synthesises → Answer + citations
```

Three skills drive the entire workflow. Python scripts are thin wrappers that handle file I/O; Claude does all understanding.

## Usage — the three-skill workflow

### Skill 1: rfp-data-prep

**When to use:** After adding new RFPs to `rfp_data/`, or to rebuild everything from scratch.

**Trigger phrase:** Say to Claude:
> "Prep my RFP data"

**What happens:**
1. `extract_text.py` runs — extracts text from every file in `rfp_data/`
2. Claude reads each extracted text file and extracts entities and relationships into `graph/entities.json` and `graph/relationships.json`
3. `build_graph.py` runs — builds the graph and detects communities
4. `generate_graph_html.py` runs — produces `graph/knowledge_graph.html`

**Expected output:**
```
Data prep complete.
  Documents processed : 2
  Entities extracted  : 59  (16 standards, 9 locations, 7 concepts, ...)
  Relationships       : 74
  Communities found   : 8
  Graph visualisation : graph/knowledge_graph.html
```

**Time estimate:** ~5–10 minutes for 10 documents, depending on document length.

---

### Skill 2: rfp-community-summarizer

**When to use:** After `rfp-data-prep` completes. Must run before global queries work well.

**Trigger phrase:**
> "Summarize the communities" or "Generate community summaries"

**What happens:**
Claude reads `community_map.json`, gathers entity details and source chunks for each community, and writes a structured markdown summary to `graph/communities/community_NN.md`.

**Expected output:**
```
Community summaries written.
  Communities summarized: 8
  Files written: graph/communities/community_00.md … community_07.md
```

**Why this matters:** Community summaries are what power cross-corpus queries ("which RFPs…", "compare across…"). Without them, global queries fall back to entity-only search, which misses thematic patterns.

---

### Skill 3: rfp-query-agent

**When to use:** Any time you want to ask a question about the ingested RFPs.

**Trigger phrase:** Just ask your question naturally.

> "What are Halcyon's lenders?"
> "Which RFPs require IFRS 15 revenue recognition?"
> "Compare ESG assurance requirements across all RFPs"
> "What audit deliverables does Meridian require?"
> "Which clients are private equity backed?"

**What happens:**
1. `query_graph.py` runs with your question
2. Claude reads the structured context (entities, relationships, community excerpts, source chunks)
3. Claude writes a prose answer with page-level citations

**Example answer format:**
```
Halcyon Agri-Industrial Group has three lenders under its SGD 350M
syndicated revolving credit facility:

- DBS Bank
- OCBC Bank
- Standard Chartered Bank

The facility is used to support working capital across Halcyon's 7-country
agri-industrial operations.

Source: RFP_External_Audit_Private_Company_KPMG_2026_1.docx, Page 1–2,
Section "Group Overview"
```

## Visualising the knowledge graph

Open `graph/knowledge_graph.html` in any browser (Chrome, Edge, Firefox). No server needed.

Features:
- Force-directed layout — nodes sized by connection count, coloured by community
- Click any node to see full entity details and all its relationships with citations
- Filter by community or entity type
- Search box to highlight specific entities
- Drag nodes to rearrange; scroll to zoom

To regenerate after adding new RFPs:
```bash
python skills/rfp-data-prep/generate_graph_html.py
```

## Adding more RFPs

1. Drop new PDF/DOCX/PPTX files into `rfp_data/`
2. Say: **"Prep my RFP data"** — Claude will process only new files (existing extractions are skipped if the output file already exists)
3. Say: **"Summarize the communities"** — re-run to incorporate new communities that may have emerged
4. Start querying with `rfp-query-agent`

> **Note on community count:** The number of communities is determined automatically by the Louvain algorithm. With 2 documents, 8 communities emerged. With 100 documents you may get 20–40. The `--resolution` flag on `build_graph.py` can be increased (e.g. `--resolution 1.5`) to produce more, smaller communities, or decreased (`--resolution 0.5`) for fewer, broader ones.

## How GraphRAG differs from traditional RAG

| | Traditional RAG | GraphRAG (this system) |
|---|---|---|
| Storage | Vector embeddings in a DB | Entities + relationships in a graph |
| Retrieval | Cosine similarity to query | Entity search + graph traversal |
| Cross-doc queries | Weak — similarity-based | Strong — community summaries |
| Citations | Chunk-level | Page + section level |
| Relationships | Not modelled | Explicit (requires, governed_by, …) |
| Global questions | Poor ("all RFPs that…") | Good (community-level summaries) |
| External services | Embedding API required | None — Claude Cowork only |

The key advantage: relationships between entities are first-class. "What lenders are involved with companies that acquired targets in Indonesia?" is a graph traversal, not a similarity search.

## Troubleshooting

**"No module named community"**
```bash
pip install python-louvain
```
The package installs as `community` but is published under the name `python-louvain`.

**extract_text.py fails on a PDF**
Some scanned PDFs contain no extractable text. Use a PDF with selectable text, or run OCR on it first (e.g. with Adobe Acrobat or `ocrmypdf`).

**entity extraction produces very few entities**
Claude may have been conservative. In the rfp-data-prep skill, you can ask Claude explicitly: "be thorough — extract every named standard, regulation, entity, and concept mentioned."

**community_map.json exists but communities/ folder is empty**
You need to run `rfp-community-summarizer` — the graph is built but summaries have not been written yet.

**Graph visualisation shows disconnected clusters**
This is expected when communities have no cross-community relationships. As more RFPs are added, cross-community edges (e.g. a standard appearing in multiple client contexts) will connect the clusters.

## Entity and relationship types reference

### Entity types

| Type | Examples |
|---|---|
| `client` | Halcyon Agri-Industrial Group, Meridian International Holdings |
| `service_provider` | KPMG |
| `service` | External Audit Services, ESG & Sustainability Assurance |
| `investor` | Halcyon Capital Partners LP |
| `standard` | IFRS 15, ISA 315, ISQM 1, SOX 404(b), ISSB S1 |
| `regulator` | PCAOB, FRC, SEC, ACRA |
| `location` | Singapore, United Kingdom, Indonesia |
| `concept` | Revenue Recognition, Business Combination Accounting, GHG Emissions |
| `lender` | DBS Bank, OCBC, Standard Chartered |
| `financial_instrument` | SGD 350M Syndicated Revolving Credit Facility |
| `acquisition_target` | PT Agro Nusantara Jaya, Vietnam Fresh Logistics JSC |
| `technology` | SAP S/4HANA, KPMG Clara, Kyriba |
| `exchange` | London Stock Exchange, NYSE |
| `deliverable` | Management Letter, Covenant Compliance Confirmation |

### Relationship types

| Type | Meaning |
|---|---|
| `requires` | Entity A requires service/standard/entity B |
| `issued_by` | RFP issued by client to service provider |
| `owned_by` | Client owned by investor |
| `governed_by` | Entity governed by regulator or standard |
| `located_in` | Headquarters location |
| `operates_in` | Country of operation |
| `has_lender` | Client's lending relationship |
| `acquired` | M&A — entity acquired target |
| `uses` | Technology or tool usage |
| `requires_audit_focus` | Audit scope complexity area |
| `mentions` | Document mentions entity (weaker link) |
| `has_deliverable` | Service produces deliverable |
| `listed_on` | Company listed on exchange |
| `has_instrument` | Client has financial instrument |
| `similar_to` | Entities with overlapping scope |
| `part_of` | Subsidiary / divisional relationship |
| `has_budget` | Service or engagement has an indicative fee / budget amount |

## Contributing and extending

To add new entity types or relationship types, edit `skills/rfp-community-summarizer/SCHEMA.md` and update the type lists in the `rfp-data-prep` skill's SKILL.md. The Python scripts are schema-agnostic — they treat entities and relationships as generic JSON objects.

To add a new script (e.g. for deduplication, export to CSV, or Neo4j ingestion), place it in `scripts/` and reference it from the relevant skill's SKILL.md.
